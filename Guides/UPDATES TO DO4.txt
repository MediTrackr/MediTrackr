### 2. Client-Side Math Risks
*   **The Problem:** Floating-point arithmetic in JavaScript can lead to precision errors (e.g., `0.1 + 0.2 !== 0.3`)[cite: 14]. Using `.reduce` directly on raw numbers for financial totals like `netIncome` can occasionally result in tiny decimal discrepancies[cite: 14].
*   **The Fix:** For high-accuracy reporting, it is safer to perform these calculations in cents (integers) and convert to decimals only for the final `.toFixed(2)` display[cite: 14].

*   **The Fix:** Standardize your date strings to include a specific time (e.g., `T00:00:00Z` and `T23:59:59Z`) to ensure the Supabase filter captures the full 24-hour window regardless of local timezones.

### 1. Brittle CSV Parsing

If a user includes a comma within a description or vendor name (e.g., "Amazon, Inc."), the columns will shift, and the "Amount" field will likely end up containing a string, causing the `parseFloat` to return `NaN`[cite: 15].
*   **The Fix:** Use a dedicated parsing library like `PapaParse` or implement a regex-based split that respects quoted strings to ensure data remains in the correct columns.

### 2. State-Input Sync Lag
*   **The Problem:** Because the `rows` state is an array of objects, every keystroke in a description or amount field triggers a `map` function to update the state[cite: 15]. While fine for 2ā€“3 rows, if a user imports a CSV with 100+ rows, the UI will become noticeably laggy as the entire list re-renders on every character typed[cite: 15].

### 3. Date Formatting Inconsistency
*   **The Problem:** The code generates dates using `.toISOString().split('T')[0]`, which is in UTC[cite: 15]. If a user in Montreal (EST) adds an expense at 10:00 PM on May 3rd, the code might default the date to May 4th[cite: 15].
*   **The Fix:** Use local date construction (e.g., `new Date().toLocaleDateString('en-CA')`) to ensure the default date matches the user's actual calendar day.

*   **The Problem:** The `records` object being sent to Supabase uses `parseFloat(r.amount)`[cite: 15]. If the amount input contains invalid characters (like a `$` sign), `parseFloat` might return `NaN`, which PostgreSQL will reject as a numeric violation, causing the entire batch to fail[cite: 15].
*   **The Fix:** Sanitize the input by removing non-numeric characters before parsing: `parseFloat(r.amount.replace(/[^0-9.]/g, ''))`.

### 6. Mode Switch Data Loss
*   **The Problem:** Switching between "Manual Entry" and "CSV Import" does not clear the `rows` state[cite: 15]. While this allows users to add to their list, if they import a bad CSV and then try to switch back to manual to fix it, the "dirty" data remains in the state[cite: 15].
